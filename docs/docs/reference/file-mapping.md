# File Mapping

The [FILE_MAPPING.md](https://github.com/blaze-commerce/blaze-dev-kit/blob/main/FILE_MAPPING.md) in the repository root is the **single source of truth** for every file in the Blaze Dev Kit.

## Purpose

The file mapping serves three critical functions:

1. **Deploy Verification** - `verify-deploy-zip.sh` validates that the release zip matches the `[DEPLOY]` files listed in FILE_MAPPING.md
2. **PR Enforcement** - `verify-pr.yml` CI check rejects PRs that place files in the wrong directory
3. **Version QA** - When auditing a site installation, compare against FILE_MAPPING.md to detect missing or extra files

## Categories

| Label | Directory | Destination |
|-------|-----------|-------------|
| `[DEPLOY]` | `deploy/` | Kinsta sites (via release zip) |
| `[DOCS]` | `docs/` | GitHub Pages (MkDocs Material) |
| `[QA]` | `qa/` | Local Playwright tests (never deployed) |
| `[REPO]` | Root, `.github/`, `scripts/`, `eval/` | Repository only (never deployed) |

## Auto-Update

FILE_MAPPING.md is automatically regenerated by the `update-mapping.yml` GitHub Action on every merge to `main`. This ensures it never goes stale.

## Using FILE_MAPPING.md

### As a Developer

Before adding a new file, check FILE_MAPPING.md to understand:
- Which directory it belongs in
- What category it falls under
- Whether a similar file already exists

### As a Maintainer

When reviewing PRs, use FILE_MAPPING.md to verify:
- New files are in the correct directory
- Removed files are reflected in the mapping
- No files are in unexpected locations

### For Site Auditing

Compare a site's `.claude/` contents against the `[DEPLOY]` section to find:
- Missing files that should be present
- Extra files that shouldn't be there
- Version mismatches
